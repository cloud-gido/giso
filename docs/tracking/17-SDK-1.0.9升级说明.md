# SDK 1.0.9 升级说明（1.0.8 → 1.0.9）

> **历史版本说明**：1.0.9 已发布；应用级心跳口径在 **1.0.10** 修正，新接入请使用 [1.0.10 心跳修复说明](18-SDK-1.0.10心跳修复说明.md)。
> **读者**：需要了解 **1.0.8 → 1.0.9** 历史变化的外部 App 团队。
> **原则**：接入方式、App Key、endpoint、登记表 **不变**；**只改依赖版本号**。  
> 详细 FAQ：[08-接入常见问题FAQ](08-接入常见问题FAQ.md)（Q21–Q25）。

## 二、1.0.9 相对 1.0.8 改了什么

### 新增（SDK 自动，业务不用改代码）

| 项 | 说明 |
|---|---|
| **`app_heartbeat`** | 第 10 个标准事件；**仅前台**默认每 60s 上报，退后台补最后不足一个周期的区间后停止；只带本区间 `page.fg_dur`，不继承页面 |
| **远程配置** | `GET /v1/config` 增加 `heartbeat_interval_ms`（15s–300s，默认 60000） |

### 修复（SDK 内部，业务不用改代码）

| 项 | 说明 |
|---|---|
| **退后台发送** | 切后台尽量排空队列（Android 同步 flush + WakeLock） |
| **冷启动顺序** | 磁盘续传延后到首次 `app_foreground` 之后，避免历史事件插在 `app_launch` 前面 |
| **回前台顺序** | 先清残留再发 `app_foreground` |

### 1.0.8 已有、1.0.9 继续用

- **`biz_did`**：`setBizDid()` / `clearBizDid()`，无账号历史设备 ID 去重
- **`sdk_runtime`**：区分 `native` / `flutter` / `web`

## 三、常见误解（升级前必读）

### 1. `app_launch` 和进前台是不是重复？

**不是。**

| 事件 | 含义 |
|------|------|
| `app_launch` | **进程冷启动**（被杀死后重新打开 App） |
| `app_foreground` | 进程还在，从后台回到前台 |

若每次「退出再进」都看到 `app_launch`，多半是 **系统杀了进程**（OEM 常见），不是 SDK 重复上报。

### 2. SDK 会不会清我们业务数据？

**不会。** SDK 只读写自己的键（如 `giso_tracker` 下的 `did`）和上报队列，**不会**清除业务的 SharedPreferences、数据库或登录态。

### 3. `app_launch` / `app_install` 会不会改 `did`？

**都不会改写 `did`。**

- `did` 变：仅 **首次安装**、**清数据**、**卸载重装**
- `app_install`：本地首次激活标记时打一次，常与新 `did` **同时出现**，但事件本身不生成 `did`

### 4. 没有账号怎么算新用户？

- **推荐口径**：`app_install` 或该 `did` **首次出现**（安装级新用户；**卸载重装会再算一次**）
- **跨卸载识别**：业务自管 `biz_did`（`setBizDid`）；若也只存本地，卸载同样会丢

### 5. `app_heartbeat` 和播放 `video_play_heartbeat` 一样吗？

**不一样。**

| | `app_heartbeat` | `video_play_heartbeat` |
|---|---|---|
| 类型 | **标准事件**，SDK 自动 | **业务事件**，播放器层 `bizEvent` |
| 范围 | 应用前台活跃 | 单次播放进度 |

**禁止**业务手造 `app_heartbeat` 或自造 `biz_event(app_heartbeat)`。

## 四、升级后怎么验

1. 依赖升到 **1.0.9**，`common.sdk_vrsn` 应为 `1.0.9`
2. 管理台 **实时联调**，按本机 `did` 过滤
3. 前台停留 ≥ 心跳间隔（默认 60s，或 `/v1/config` 下发值）→ 应出现 **`app_heartbeat`**，校验为「正常」
4. Home 退后台 → 心跳停止；再回前台 → 重新计时
5. 仅切后台再回（进程未杀）→ 只有 `app_foreground`，**不应**再有 `app_launch`
