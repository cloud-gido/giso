# 流程治理与质量保障

> 对应大同「规范登记、流程管理、版本控制、实时联调、上报监控」五件套，
> 按创业公司规模裁剪：用 Git/CI 替代管理平台，语义不减，成本最低。

---

## 1. 角色与职责

| 角色 | 职责（对应大同角色职责页） |
|---|---|
| 业务产品 | 在任务系统（Jira/TAPD/飞书项目）提数据需求，写清楚**要回答什么问题** |
| 数据负责人 | 把业务需求翻译成 页面/元素/参数/事件 的注册表变更（PR）；确认口径；验收数据 |
| 终端开发 | 按注册表生成的常量开发埋点；自测；联调；上线后确认 |
| 后端开发 | 服务端事实流事件开发；提供校验接口；与客户端意图流对账 |
| 测试 | 按需求进行上报测试；维护 UI 自动化校验用例 |

创业公司一人可兼多角色，但**「登记 PR 必须有数据负责人 + 终端负责人双 Review」**这条不可省——这就是流程卡点的最小实现。

## 2. 需求到上线的固化流程

```
① 提出业务需求（任务单）            业务产品
② 登记上报需求（schema PR，关联任务单）  数据负责人
③ 登记页面结构体（PR 中确认元素层级/参数继承关系）  终端开发 + 数据确认
④ 开发 & 自测（引用生成的常量；本地 debug 模式实时看上报）  终端/后端
⑤ 测试上报数据（联调环境实时校验）       测试
⑥ 上报验收（数据负责人在验收看板确认量级和参数完整性，任务单流转为完成）
⑦ 版本上线 → 线上确认（按版本对比上报质量，监控 + 预警）
```

与版本发布绑定：**步骤⑥未通过，该需求不准随版本上线**（对应大同「数据产品验收上报通过，终端才可发布上线」）。

## 3. 实时联调（开发/测试期）

P0 用最薄的实现满足「实时看到自己设备的上报」：

1. SDK 提供 debug 模式：`Tracker.setDebug(true)` 后，事件附加 `debug_did` 标记并实时（不攒批）上报。
2. 网关把 debug 事件额外写入 `events_debug` topic，保留 24h。
3. 一个极简联调页（或直接 Doris 查询模板）：按 `did` 过滤，实时滚动展示事件流 + 按注册表的校验结果（缺参标黄、未登记标红、类型错标红）——对应大同实时校验的「日志明细 + 统计报告」。
4. 校验规则直接从 `schema/*.yaml` 加载，注册表更新即生效，不存在第二份口径。

## 4. 接入层校验（线上常态）

每条事件在网关按注册表校验，结果分三类：

| 结果 | 处理 |
|---|---|
| 正常 | 进 `giso_events_raw` |
| 缺失（必携参数为空） | 进 `giso_events_raw` 并打 `_quality=missing` 标记，计数 |
| 错误（未登记事件/元素/页面、类型错误） | 进 `giso_events_quarantine` 隔离区，计数 |

隔离区数据保留 7 天可回放——修复 schema 或客户端后可重新接入，不丢数据。

## 5. 上报监控与告警（线上）

| 监控项 | 规则示例 |
|---|---|
| 上报量级 | 核心事件（page_enter、element_click、bet_placed）按版本环比，波动 > 30% 告警 |
| 缺参率 / 错误率 | 按事件×参数维度，错误率 > 1% 告警（对应大同校验明细表：上报量/异常量/缺失量/错误量） |
| 新版本灰度 | 新版本发布后 24h 内单独看新版本上报质量，异常即停止放量 |
| 双链路对账 | bet_submit vs bet_placed+bet_rejected、pm_order_submit vs filled+cancelled，按 ID 日对账 |
| 链路存活 | 网关 QPS、Kafka lag、Doris Routine Load 延迟，基础设施告警 |

## 6. 版本控制

- 注册表带 `since` / `deprecated` 字段，事件结构变更必须新增字段而不是改义。
- 校验按客户端 `app_vrsn` 匹配对应注册表版本（Git tag），新老版本结构体并存校验，对应大同 v1.0/v2.0 并行校验设计。
- 每次发版生成「本版本埋点变更清单」（从 schema 的 git diff 自动产出），数据团队据此更新看板。

## 7. 工具演进（P1/P2）

- P1：联调控制台 Web 化；质量看板（质量统计/排行/趋势）；测试侧 UI 自动化遍历核心页面比对上报快照。
- P2：注册表迁入管理后台（登记/审批/版本控制 UI 化），打通任务系统自动关联；可视化标记辅助登记元素；数据血缘（事件 → 表 → 看板）。
