# GISO · 玑源 · 开源产品全景方案

> **产品名**：GISO（玑源）— **GIDO 的数据源头**，与母品牌「GIDO」关联：GISO = **G**i **I**s the **S**ource **O**f data（数据源头）。
>
> **定位**：Schema 驱动的移动端 + Web 行为分析平台。

---

## 1. 产品边界

### 1.1 做什么（In Scope）

| 能力域 | 说明 |
|---|---|
| **采集** | Web / Android / iOS / **Flutter** / Server 五端 SDK，9 个标准事件，参数继承，曝光口径统一 |
| **治理** | 参数池 / 页面池 / 元素池 / 业务事件注册表；PostgreSQL 运行时权威；CI 校验 + 五端常量生成；管理台 CRUD / 审批 / 批量 / CSV 导入 |
| **接入** | 网关鉴权、限流、Schema 校验、raw/quarantine 分流、远程配置下发 |
| **管道** | 可插拔 Sink（File / Kafka）；Kafka spill 兜底 + 自动回放 |
| **落地** | Doris ODS + Routine Load；DWD 视图 + ADS 看板 SQL |
| **质量** | 实时联调、用例断言、三维度质量报表、Prometheus 指标、对账脚本 |
| **分析** | Metabase 接入；预置 DAU/留存/CTR/漏斗/完播/实验分桶 SQL |

### 1.2 不做什么（Out of Scope · 可插件扩展）

- 可视化圈选登记（**v2 已落地**：管理台 Visual Picker + ViewTree SDK + visual-draft API）
- 全功能 AB 实验平台（通过 `pt` 透传与外部实验系统对接）
- 实时 OLAP 替代 Doris（Sink 接口可接 ClickHouse / BigQuery）
- 用户画像 / ID-Mapping 中台（仅提供 did/uid 字段，映射由业务 DW 完成）

## 2. 五层架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│  L5 分析应用层   Metabase / Superset / 自研报表 / 指标 API              │
├─────────────────────────────────────────────────────────────────────────┤
│  L4 数据仓库层   Doris ODS → DWD 视图 → ADS 看板 SQL                    │
├─────────────────────────────────────────────────────────────────────────┤
│  L3 消息管道层   Kafka (giso_events_raw / giso_events_quarantine)                 │
├─────────────────────────────────────────────────────────────────────────┤
│  L2 接入治理层   Gateway：校验 · 鉴权 · 限流 · 指标 · 管理台 · 配置下发   │
├─────────────────────────────────────────────────────────────────────────┤
│  L1 采集 SDK层   Web · Android · iOS · Flutter · ServerReporter          │
└─────────────────────────────────────────────────────────────────────────┘
         ▲                              ▲
         │ schema/*.yaml (Git)          │ tools/codegen (CI)
         └──────── 注册表（唯一事实源） ──┘
```

**数据流（Happy Path）**

```
SDK 攒批 → POST /v1/track → Registry 校验
  ├─ ok/missing → giso_events_raw → Routine Load → ods_events
  └─ error      → giso_events_quarantine → ods_events_quarantine（可回放）

ServerReporter（资金事实）→ Kafka giso_events_raw（绕过网关，同信封同 topic）
```

**控制流**

```
需求 → 改 schema/*.yaml → PR 评审 → CI(generate.py --check) → 合并
     → 三端引用生成常量开发 → 管理台联调 → 断言 API 回归 → 发版
```

## 4. 核心设计决策（开源版须保持兼容）

### 4.1 事件模型：9 事件 × 注册表

不允许业务随意 `track('my_custom_event')`。全端只有 9 个 `event`，差异用 `pgid / eid / biz.code + params` 表达。这是控制事件膨胀、保证 CTR/漏斗口径一致的唯一可持续做法。

### 4.2 双事实源：client 行为 + server 资金

| 类型 | 上报方 | 示例 | 网关策略 |
|---|---|---|---|
| 行为流 | 端上 SDK | 曝光、点击、播放心跳、`bet_submit` | 走网关校验 |
| 事实流 | `sdk/server` | `bet_placed`、`deposit_completed` | 直写 Kafka；端上冒充 → error |

### 4.3 校验三分类

| 结果 | 去向 | 含义 |
|---|---|---|
| ok | raw | 完全合规 |
| missing | raw + `_quality=missing` | 必携参数空，可分析但需修复 |
| error | quarantine + `_issues` | 未登记/结构体违规/事实流冒充，不入仓 |

### 4.4 可插拔 Sink

```yaml
sinks: [file]           # 本地开发
sinks: [kafka]          # 生产
sinks: [file, kafka]    # 迁移期双写
```

新增 Sink（如 S3、ClickHouse）实现 `EventSink` 接口即可，不改校验逻辑。

## 5. 部署模式

### 5.1 模式 A：本地开发（零依赖）

```bash
cd server/gateway && mvn package -DskipTests
java -jar target/giso-gateway.jar --config gateway.yaml
# sinks: [file] → ./data/*.jsonl
# 管理台 http://localhost:8080/admin/
```

适合：SDK 开发、注册表调试、联调断言。

### 5.2 模式 B：Docker Compose（推荐体验）

```bash
cd deploy && docker compose up -d
# gateway:8123 + kafka:9092 + metabase:3000
```

适合：Demo、集成测试、小团队 PoC。Doris 仍建议独立集群（见 `server/doris/`）。

### 5.3 模式 C：生产（Kafka + Doris + K8s）

```
                    ┌─ Gateway Pod ×N（无状态）
  SDK ──► LB ───────┤     注册表 ← PostgreSQL（RDS 库 giso）
                    └─ LISTEN/轮询 多副本热更新
                           │
                           ▼
                      Kafka Cluster
                           │
                           ▼
                      Doris FE/BE + Routine Load
                           │
                           ▼
                      Metabase / 自研 BI
```

**生产 checklist**

- [ ] RDS 库 `giso` + Doppler `INFRA_GISO_DB_*`
- [ ] `auth.app_keys` + `admin_user/viewer_user`
- [ ] `limits.rate_limit_rps: 100`
- [ ] Kafka `acks=all` + 多副本
- [ ] Doris Routine Load RUNNING
- [ ] Prometheus 抓取 `/metrics` + `server/ops/prometheus-rules.yml`
- [ ] 注册表导出 CronJob / `tools/registry/export_yaml.py`

## 7. 与主流开源方案对比

| 维度 | GISO 玑源 | PostHog | Snowplow | Matomo |
|---|---|---|---|---|
| 自托管 | ✅ | ✅ | ✅ | ✅ |
| 移动端 SDK | ✅ 三端 | ✅ | ✅ | ✅ |
| **Schema 强校验** | ✅ 网关拦截 | 弱（后置清洗） | ✅ Enrichment | 弱 |
| **事件收敛** | ✅ 9 事件 | ❌ 自由事件 | 需自建规范 | ❌ |
| **注册表驱动** | ✅ Git+CI | 无 | Schema Registry | 无 |
| 实时联调 | ✅ 管理台 | ✅ Live events | ❌ | ❌ |
| 分析 UI | Metabase（外接） | ✅ 内置 | ❌ 需接 BI | ✅ |
| 适合场景 | **强规范多业务线 App** | 产品分析全家桶 | 数据工程管道 | Web 分析 |

**选型建议**：需要「大同级规范 + 自托管 + 博彩/交易事实流」→ **GISO 玑源**；需要「开箱即用产品分析 + Session Replay」→ PostHog；需要「纯管道 + 自建模型」→ Snowplow。
