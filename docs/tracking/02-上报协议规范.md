# 上报协议规范 v1.0

> 本协议为 App / Web / 服务端三端统一协议。任何端任何事件都使用同一个信封结构。

---

## 1. 传输层

| 项 | 规范 |
|---|---|
| 接口 | `POST https://t.{domain}/v1/track` |
| 内容 | JSON 数组（批量），`Content-Encoding: gzip`，单批 ≤ 50 条或 ≤ 256KB |
| 鉴权 | Header `X-App-Key`（白名单校验，无效返回 `401`）；sendBeacon 等无法带 Header 的场景走 query `?k=`；服务端事实流另走内网 Kafka 直写 |
| 响应 | `204` 成功；`401` 鉴权失败不重试；`413` 超体积不重试；`429` 限流**需退避重试**；其余 `4xx` 不重试（数据有误，丢隔离区）；`5xx`/超时 客户端指数退避重试 |
| 防护 | 单请求解压后 ≤ 1MB（超出 `413`）；单 IP 令牌桶限流（超出 `429`） |
| 触发上报 | 满 20 条 / 距上次 15s / 进后台或页面卸载（flush），三者先到先发 |
| 本地缓存 | 失败落盘（App: 文件队列；Web: localStorage），上限 500 条，FIFO 淘汰 |
| 时钟 | 客户端带本地时间 `ctime`，网关写入服务端时间 `stime`，分析以 `stime` 为准、`ctime` 用于排序 |

## 2. 事件信封（Envelope）

每条事件 = **公共参数（SDK 自动采集） + 事件四元组 + 业务参数**。

```json
{
  "event": "element_click",          // 9 个标准事件之一
  "log_id": "8f2c...-uuid",          // 事件唯一ID，幂等去重用
  "ctime": 1770000000123,            // 客户端毫秒时间戳

  "common": {                        // ── 公共参数，SDK 自动填充 ──
    "app_id": "giso",
    "app_pkg": "com.example.myapp",  // Android applicationId / iOS Bundle ID，SDK 自动采集
    "platform": "android",           // android | ios | web
    "app_vrsn": "1.4.0",             // 应用/前端版本
    "sdk_vrsn": "1.0.6",
    "did": "a1b2c3...",              // 设备ID（App: 自生成持久化UUID；Web: 一方cookie）
    "uid": "10086",                  // 登录用户ID，未登录为空
    "session_id": "s-9f8e...",       // 会话ID，前后台间隔>30min 重开
    "channel": "google_play",        // 安装/投放渠道
    "os_vrsn": "14",
    "dev_brand": "OPPO",
    "dev_model": "OPPO A77",
    "screen_res": "1080x2400",
    "net_type": "wifi",
    "lang": "zh-CN",
    "tz": "+08:00"
  },

  "page": {                          // ── 页面上下文，SDK 维护 ──
    "pgid": "video_detail",          // 当前页面，必须在页面池登记
    "pg_params": { "vid": "v123" },  // 页面级参数
    "ref_pgid": "rec_feed",          // 来源页面
    "ref_eid": "video_card",         // 来源点击元素（归因链）
    "pg_stay": 4520                  // 仅 page_exit：停留毫秒
  },

  "element": {                       // ── 仅元素事件 ──
    "eid": "like_btn",               // 必须在元素池登记
    "mod": "video_card",             // 所属模块（父元素）
    "pos": 3,                        // 列表位置，0-based
    "params": { "vid": "v123" },     // 继承+自有参数（SDK 自动合并）
    "exp_dur": 1200,                 // 仅 exposure：曝光时长ms
    "exp_ratio": 0.85                // 仅 exposure：最大可视比例
  },

  "biz": {                           // ── 仅 biz_event ──
    "code": "video_play_heartbeat",  // 必须在事件字典登记
    "params": { "vid": "v123", "play_dur": 30000, "is_auto": 1 }
  },

  "pt": {                            // ── 后台参数透传包（可选）──
    "rec_trace_id": "tr-8f3a",       // 推荐 trace、赔率版本、盘口快照等
    "odds_vrsn": "20260611-03"       // 服务端接口下发，端上原样透传，不参与校验
  }
}
```

字段精简规则：`element` 仅元素事件携带，`biz` 仅业务事件携带；空对象不传。

**后台参数透传（`pt`）**：服务端才知道的口径（推荐 trace_id、赔率版本、盘口快照、实验分组）由业务接口随响应下发，端上**不理解内容**、不登记、不校验，原样透传，落 Doris `pt` JSON 列。SDK 三端 API 一致：`enterPage(pgid, params, pt)` 页面级（本页所有事件携带）、`bind(el, {.., pt})` 元素级（随继承链合并，子级覆盖父级）、`bizEvent(code, params, pt)` 调用级。归因 JOIN 示例：`SELECT ... FROM ods_events WHERE pt->'$.rec_trace_id' = ...`。

## 3. 触发时机精确定义（口径的根）

> 大同经验：口径不统一的根源是触发时机不统一。以下定义内置于 SDK，业务方不可自定义。

| 事件 | 精确定义 |
|---|---|
| `page_enter` | App：Activity/Fragment `onResume`；Web：路由挂载完成且 `document.visibilityState === 'visible'` |
| `page_exit` | App：`onPause`；Web：路由卸载或 `pagehide`。携带 `pg_stay` |
| `element_exposure` | 元素进入视口面积比例 ≥ 50% 且持续 ≥ 500ms 记一次；滚出视口超过 80% 后再次满足条件可重新记。同一元素实例在同一次页面进入内最多记 3 次（防刷屏抖动）。阈值可远程配置 |
| `element_click` | 用户完成点击手势的瞬间。**不是**发起请求时、**不是**接口返回成功时。需要表达「操作成功」用 `biz_event` |
| `app_background` | 携带 `fg_dur`（本次前台毫秒） |
| `video_play_heartbeat` | 播放中每 30s 一条 + 暂停/退出时补一条，携带累计 `play_dur`。播放时长口径 = sum(heartbeat 区间)，不信任客户端单条总时长 |

## 4. 参数命名规范

- 全小写 snake_case，长度 ≤ 32；ID 类后缀 `_id`（`vid`、`cp_id`、`match_id`）；时长统一毫秒后缀 `_dur`；布尔用 0/1 整数。
- 金额统一**最小货币单位整数**（分/cent）+ 配套 `currency` 参数，禁止浮点。
- 任何新参数必须先在 `schema/params.yaml` 登记（key、类型、说明、取值规则、负责人），CR 通过后方可使用。

## 5. 服务端事实流

资金/结算类事件由后端服务直写 Kafka（同一信封，`platform: "server"`，`common` 仅保留 `app_id/uid/stime`）。要求：

1. 业务库事务提交成功后发出（outbox 模式），保证不丢不假。
2. `log_id` 用业务单号派生，天然幂等。
3. 客户端同名意图事件（如 `bet_submit`）与服务端事实事件（`bet_placed`）通过 `bet_id` 关联，日级对账，差异率 > 1% 告警。

## 6. 隐私与合规

- `did` 不使用系统广告标识符之外的硬件 ID；用户注销时 `uid` 关联数据按策略删除。
- 博彩/预测市场线：上报中携带 `geo_region`（服务端按 IP 解析后写入，客户端不采 GPS），用于合规审计；不采集任何身份证件、银行卡信息进事件流。
- IP 仅在网关侧用于解析地域后截断存储（/24）。
