# 埋点方案总体设计 · GISO 玑源

> **GISO**（玑源）= GIDO 的数据源头 · Gi Is the Source Of data
> 适用范围：App（Android / iOS）+ Web 前端
> 业务线：长视频、体育博彩、预测市场、资讯
> 设计参考：大同上报思想（事件收敛、参数继承、统一管理、流程闭环）

---

## 1. 我们要解决什么问题

创业公司从 0 开始做埋点，最容易掉进的坑正是文档里描述的三方痛点：

| 角色 | 痛点 | 后果 |
|---|---|---|
| 业务侧 | 数据对不上、口径不明确 | 决策不准确 |
| 终端侧 | 上报时机不一、字段命名不统一、debug 困难 | 疲于找问题 |
| 数据侧 | 字段不规范、事件维护成本高、验证周期长 | 始终算不对 |

根因是「**自由埋点**」：每个需求各自定义事件名、参数名、触发时机，事件随业务膨胀（视频点赞、帖子点赞、比赛点赞……各报各的），最终 2500+ 个小事件无人能维护。

**本方案的核心决策：从第一天起就采用「事件收敛 + 参数继承 + 注册表驱动」的半自动埋点体系，把规范固化进 SDK 和流程，而不是依赖开发自觉。**

## 2. 核心思想

### 2.1 事件收敛：小事件 => 大事件 × 元素

不允许业务随意新增事件名。全端只有 **9 个标准事件**，全部由 SDK 控制触发时机：

| 事件 | event_code | 触发时机（SDK 统一定义） | 端 |
|---|---|---|---|
| 首次激活 | `app_install` | 首次启动且本地无激活标记时 | App |
| 应用启动 | `app_launch` | 进程冷启动完成 | App |
| 进前台 | `app_foreground` | 应用切换到前台 | App |
| 退后台 | `app_background` | 应用切换到后台（携带本次前台时长） | App |
| 页面进入 | `page_enter` | 页面对用户可见（onResume / route mounted） | App + Web |
| 页面退出 | `page_exit` | 页面不可见（携带页面停留时长） | App + Web |
| 元素曝光 | `element_exposure` | 元素可视面积 ≥ 50% 且持续 ≥ 500ms（可配置） | App + Web |
| 元素点击 | `element_click` | 用户按下并完成点击手势（不是发请求、不是返回成功） | App + Web |
| 业务事件 | `biz_event` | 不可被曝光/点击建模的领域事件（见 2.3） | App + Web + Server |

业务差异全部通过 **元素（element）+ 参数（params）** 表达：

```
以前：video_like_click、post_like_click、match_like_click……（事件膨胀）
现在：element_click + eid=like_btn + 继承所在页面/卡片的参数
```

点赞 CTR = `element_exposure(eid=like_btn)` 分之 `element_click(eid=like_btn)`，口径天然统一，因为分子分母的触发时机都由 SDK 控制。

### 2.2 参数继承：页面 → 模块 → 元素

参数沿可视结构自动向下继承，子元素不需要重复填充：

```
推荐页 (page: rec_feed)
 └─ 视频卡 (mod: video_card, params: vid)
     ├─ 播放按钮 (eid: play_btn)        → 上报时自动带上 vid
     ├─ CP头像   (eid: cp_avatar, params: cp_id) → 自动带上 vid + cp_id
     ├─ 点赞按钮 (eid: like_btn)        → 自动带上 vid
     └─ 分享按钮 (eid: share_btn)       → 自动带上 vid
```

终端开发只需要在「视频卡」这一层 set 一次 `vid`，下面所有元素的曝光/点击自动携带。同一个元素（如 `video_card`）出现在推荐页、精选页、搜索结果页时，参数定义完全一致，只是 `pgid` 不同。

### 2.3 业务事件：客户端行为流 + 服务端事实流双链路

曝光/点击覆盖不了的领域事件收敛为少量 `biz_event`，并且**凡是涉及钱和结算的事件，以服务端上报为唯一事实源**，客户端同名事件仅用于行为漏斗分析：

| 域 | 事件示例 | 事实源 |
|---|---|---|
| 播放 | `video_play_start` / `video_play_heartbeat` / `video_play_end` | 客户端 |
| 博彩 | `bet_submit`（客户端意图）/ `bet_placed`、`bet_settled`（服务端事实） | 服务端 |
| 预测市场 | `order_submit`（客户端）/ `order_filled`、`market_resolved`（服务端） | 服务端 |
| 账务 | `deposit_completed` / `withdraw_completed` | 服务端 |

这是博彩/交易类产品的强制要求：GGR、流水、结算对账绝不能依赖客户端数据；客户端数据只回答「用户在哪一步流失」。

### 2.4 注册表驱动（参数池 / 元素池 / 页面池）

所有页面、元素、参数、业务事件先登记后使用，注册表是唯一事实源（见 `schema/` 目录）：

- **参数池** `params.yaml`：参数 key、类型、取值规则全局唯一。`vid` 在任何端任何页面都叫 `vid`，类型都是 string。
- **元素池** `elements.yaml`：元素 eid、所属业务域、必携参数。`like_btn` 全局只有一个定义。
- **页面池** `pages.yaml`：页面 pgid、页面参数、所属业务线。
- **事件字典** `biz_events.yaml`：业务事件 code、参数、事实源（client/server）。

注册表的三个用途：
1. **代码生成**：生成 TS 常量 / Java 常量，开发引用常量而不是手写字符串，写错直接编译失败。
2. **接入校验**：数据接入层按注册表做 schema 校验，未登记事件 / 缺参 / 类型错误的数据进隔离区并告警。
3. **元数据即文档**：数据分析直接读注册表理解口径，不用翻代码、问开发。

创业阶段不需要先建管理后台，**Git 仓库里的 YAML + Code Review 就是登记和审批流程**。规模上来后再把注册表搬进管理平台（演进路线见 §5）。

## 3. 总体架构

```
┌─────────────────────────── 终端 ───────────────────────────┐
│  Android SDK (Java)   iOS SDK    Web SDK (TS)    服务端SDK   │
│  · 公共参数自动采集    · 页面/元素声明式埋点                   │
│  · 曝光/点击时机统一    · 本地缓存 + 批量上报 + 失败重试        │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTPS  POST /v1/track (batch, gzip)
┌──────────────────────────▼──────────────────────────────────┐
│  接入层  Ingestion Gateway（出口插拔式，gateway.yaml 配置）     │
│  · 解包/解压/限流  · 按注册表 schema 校验  · SDK 远程配置下发   │
│  · 合法 → sink: events_raw（kafka / file，可双写）             │
│  · 非法 → sink: events_quarantine（隔离区 + 告警）             │
│  · kafka 不可用 → 本地 spill 兜底，恢复后回放                   │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│  数据层                                                       │
│  Kafka → Doris Routine Load → ods_events（明细 + 实时查询）    │
│        → S3/OSS 冷备（原始日志，重算兜底）                      │
│  服务端事实流（bet_placed 等）由后端服务直写 Kafka               │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│  应用层                                                       │
│  · Superset/Metabase 看板    · 实时联调控制台（debug 模式查询） │
│  · 质量监控（上报量/缺参率/错误率告警）  · 下游：推荐、风控、CRM  │
└──────────────────────────────────────────────────────────────┘
```

**自研 vs 采购的判断**（对应文档中的市场方案对比）：

| 方案 | 结论 |
|---|---|
| GrowingIO / MixPanel / 神策 | 按量计费，博彩类目存在合规与数据主权风险；可视化埋点不适合我们复杂场景 |
| Firebase | 字段数/长度受限，数据出境，博彩类目政策风险高，不可控 |
| **本方案：自研轻协议 SDK + 开源管道（Kafka + Doris）** | 协议和数据完全自主，成本是固定的机器成本；SDK 复杂度被「9 事件收敛」大幅压低，创业团队 2~3 人月可落地 |

> 折中选项：行为分析看板初期可以先用自部署 PostHog 吃 `events_raw`，省掉看板开发；协议与 SDK 仍按本方案自建，保证随时可替换。

## 5. 演进路线

| 阶段 | 目标 | 内容 |
|---|---|---|
| P0（第 1~2 月） | 能采、能查、能信 | 协议 + 双端 SDK + 接入校验 + Kafka→Doris 明细 + 注册表（Git 管理）+ 核心漏斗看板 |
| P1（第 3~6 月） | 流程闭环 | 实时联调控制台、上报质量监控告警、UI 自动化校验接入回归、服务端事实流对账 |
| P2（6 月+） | 平台化 | 注册表 Web 化（登记/审批/版本控制）、可视化圈选辅助登记、数据血缘、A/B 实验分流参数下发 |

P0 原则：**规范先行，平台后置**。大同体系里管理平台是几十人团队的产物，创业公司先用 Git + CI 承载同样的「登记-评审-版本控制」语义，成本接近零，但规范语义一天都不缺位。
