总体结论 大同方法论的核心三件事——事件收敛、结构即约束（登记+强校验）、客户端/服务端事实分流——已全部落地， 部分实现（端上冒充服务端事实的网关硬拦截、Kafka spill 自动回放）比宣讲材料更严格。 差距集中在平台化工程：环境分流、覆盖率监控、版本化校验、口径中心。下方第一优先三项合计约两周可补最痛盲区。

能力矩阵

改善路线（按投入产出排序）

第一优先 · 约 2 周

埋点覆盖率报表（登记未上报）约 1 周

为什么重要：目前只能发现「上报未登记」，反方向漏埋/失效要等业务看数才暴露，是质量监控最大盲区。

建议做法：网关统计实报 pgid/eid/code 集合，与注册表求差集；管理台加「登记未上报清单」（status=live），作为发版前检查项。

测试 / 生产环境分流约 1 周

为什么重要：联调与测试流量混入生产表，污染 DAU/漏斗等核心指标。

建议做法：信封 common 增加 env 字段（debug 包默认 test）；网关按 env 路由到独立 topic/表，联调页默认只看 test 流。

需求单关联（issue_link）2-3 天

为什么重要：条目与需求来源断链，「这个埋点为什么存在」无法追溯——大同登记单的核心价值。

建议做法：四张注册表加 issue_link 字段，CI 校验 URL，管理台渲染为可点击链接。

第二优先 · 约 1 个月

版本化结构体校验约 2 周

为什么重要：改既有元素必携参数时，老版本 App 仍在上报旧结构，会被新规则误判 error。

建议做法：注册表 params 支持 since/until 版本区间，网关按 app_vrsn 选取生效规则。

隔离区修复回放工具约 1 周

为什么重要：补登记后，之前进隔离区的数据需要手工捞回，流程断在「修复后回灌」。

建议做法：ops 脚本消费 events_quarantine，剥离 _issues 后重新过网关校验，通过即写回 events_raw。

采样率与降级下发约 1 周

为什么重要：大促/热点赛事高峰曝光量级最大，目前只能整体 429 限流，无法精细降级。

建议做法：/v1/config 增加 per-event 采样率，SDK 按 did 哈希决定是否命中，信封带 sample_rate 供分析还原。

第三优先 · 规模化阶段

指标字典 / 口径中心约 1 月

为什么重要：完播率、转化率等口径活在 SQL 注释里，人员流动后口径漂移是必然。

建议做法：叠加 metrics.yaml，管理台展示，看板 SQL 由指标定义生成。

注册表 DB 化 + 域级权限约 1 月

为什么重要：YAML 单实例在网关水平扩容时写回会冲突；业务线多了需要按域授权。

建议做法：注册表迁 DB，网关启动从 DB 加载 + 变更广播；权限模型加 domain 维度。

可视化圈选登记 + UI 自动化校验1 月以上

为什么重要：对齐大同最后一块：在真实界面圈选元素完成登记、自动遍历校验上报。

建议做法：依赖 ViewTree 上报与遍历驱动，建议有专职工具团队后启动；先用断言 API 接入 Appium/Maestro 过渡。

不建议现在做的 可视化圈选登记和 UI 自动化遍历是大同投入最重的部分。当前用「常量代码生成 + 断言 API + 人工联调」已覆盖约 80% 防错价值，建议业务量级上来之后再投。
